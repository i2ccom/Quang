
That is the native RsTs advantage.

# 1. Core Concept: Dependency Value Score

Every package should receive a **Dependency Value Score**.

```txt
Dependency Value Score =
  usefulness
  - complexity cost
  - security risk
  - maintenance risk
  - duplication cost
  - runtime cost
  - AI comprehension cost
```

```txt

```

RsTs can compute this because it has a **CodeGraph**, not just file text.

# 2. Dependency Metrics

Each dependency should be measured across several dimensions.

## 2.1 Usage Surface

How much of the package do we actually use?

```txt
Package: lodash
Imported functions: debounce, cloneDeep
Total exported functions: 300+
Usage surface: low
```

Metric:

```txt
usage_surface = used_exports / total_exports
```

Example:

```txt
used_exports: 2
total_exports: 300
usage_surface: 0.0066
```

Interpretation:

```txt
Very low usage → candidate for partial import, inline, or replacement.
```

---

## 2.2 Dependency Weight

How much does this dependency bring with it?

```txt
dependency_weight =
  package_size
  + transitive_dependencies
  + install_files
  + build_scripts
  + native binaries
```

Example:

```txt
left-pad:
  code size: tiny
  transitive deps: 0
  risk: low
  but usefulness: also tiny

some-image-helper:
  code size: medium
  transitive deps: 42
  postinstall: yes
  native binary: yes
  risk: high
```

---

## 2.3 Replacement Difficulty

Can RsTs safely replace this dependency with generated or standard code?

```txt
replacement_difficulty =
  behavior_complexity
  + API surface used
  + edge case count
  + test coverage requirement
  + security sensitivity
```

Example:

```txt
is-number:
  replacement difficulty: very low

date/time library:
  replacement difficulty: high

crypto library:
  replacement difficulty: very high, do not auto-replace
```

This is important. RsTs should not aggressively replace dangerous things like cryptography, authentication, database drivers, or compression algorithms unless using vetted replacements.

---

## 2.4 Security Risk

Measure:

```txt
- maintainer reputation
- package age
- update frequency
- known vulnerabilities
- install scripts
- obfuscation
- network access
- native code
- permission request
- AI-generated unknown code
- supply-chain anomaly
```

Risk example:

```txt
Package asks for:
  fileRead: $HOME
  network: *
  postinstall script: yes

But app only uses:
  slugify(title)

Recommendation:
  replace or sandbox
```

---

## 2.5 Duplication Score

CodeGraph can detect if many packages contain similar code.

```txt
Package A includes small function normalizePath
Package B includes similar normalizePath
App has similar normalizePath
```

Fluid can propose:

```txt
Merge into shared internal utility node.
```

Metric:

```txt
duplication_score =
  semantic_similarity
  + graph_similarity
  + behavior_similarity
```

This is much better than text matching because CodeGraph can compare behavior, input/output shape, and tests.

---

## 2.6 Runtime Cost

Measure:

```txt
- cold start impact
- memory cost
- CPU cost
- bundle size
- WASM size
- parse/compile time
- dynamic import cost
```

Example:

```txt
Dependency contributes:
  3 KB actual used code
  400 KB bundled code
  1.2 MB transitive package weight
```

Recommendation:

```txt
Replace with internal node.
```

---

## 2.7 AI Comprehension Cost

This is unique to RsTs.

Some dependencies are hard for AI agents to reason about:

```txt
- dynamic runtime behavior
- monkey patching
- eval
- unclear types
- weak docs
- complex side effects
- no tests
- minified source
- undocumented exports
```

Metric:

```txt
ai_comprehension_cost =
  dynamic_behavior
  + weak_typing
  + poor_docs
  + high hidden side effects
```

A package may be technically working but bad for AI-maintained software.

# 3. Dependency Decision Classes

RsTs/Fluid can classify dependencies into several actions.

```txt
KEEP
PARTIALIZE
INLINE
MERGE
REPLACE
SANDBOX
VENDOR
ABANDON
QUARANTINE
```

## KEEP

Used heavily, trusted, maintained.

```txt
serde
tokio
zod
reactive core
database driver
crypto library
```

## PARTIALIZE

Only a small part is used.

```txt
Import only used functions.
Tree-shake package.
Fluid-pull only required graph nodes.
```

## INLINE

Very small, stable utility.

```txt
leftPad
isNumber
capitalize
clamp
range
slugify, if simple
```

## MERGE

Multiple dependencies do similar things.

```txt
merge 5 tiny string utilities into internal string-utils graph
```

## REPLACE

Use standard library or vetted RsTs/Fluid package.

```txt
replace random UUID helper with rsts:uuid
replace small retry helper with rsts:retry
```

## SANDBOX

Keep but restrict.

```txt
allow dependency to transform image
but no network
no filesystem outside temp
no secrets
```

## VENDOR

Copy into project with pinned version.

Useful when:

```txt
- package is abandoned
- code is small but important
- behavior must be frozen
- external supply-chain risk is unacceptable
```

## ABANDON

Remove completely.

```txt
dependency exists but no CodeGraph path reaches it
```

## QUARANTINE

High risk. Do not execute until reviewed.

```txt
postinstall script
obfuscated source
new maintainer
suspicious permission change
```

# 4. CodeGraph Dependency Audit

RsTs can build a full graph:

```txt
App Node
  ↓ imports
Package Node
  ↓ exports
Function Node
  ↓ calls
Permission Node
  ↓ runtime access
Data Node
  ↓ file/network/db
Risk Node
```

Then ask:

```txt
Which package functions are actually used?
Which dependencies are unreachable?
Which packages request permissions not needed by used functions?
Which transitive packages are unused after graph slicing?
Which packages duplicate internal code?
Which package update adds new risky graph edges?
```

This is the breakthrough.

Normal package managers know:

```txt
A depends on B
B depends on C
```

RsTs/Fluid knows:

```txt
A uses only function x from B.
Function x does not use C.
Therefore C can be excluded.
```

That is **graph slicing package management**.

# 5. Fluid Graph-Sliced Install

In npm, installing a package usually downloads the whole package.

In Fluid, install can become graph-sliced:

```txt
App imports:
  lodash/debounce

Fluid installs:
  debounce node
  required helper nodes
  tests/docs/types for those nodes
  no unrelated exports
```

Example:

```txt
Traditional install:
  lodash package: full

Fluid install:
  lodash.debounce subgraph only
```

This is much more powerful than tree-shaking because it happens at the **package storage/versioning level**, not only final bundle build.

```txt
+ Wduplication
+ Wruntime
+ Wai

DependencyValue =
  Wusage
+ WcomplexityProvided
+ Wstability
+ WtestCoverage
+ Wstandardness
+ WbusinessCriticality

DecisionScore =
  DependencyValue - DependencyCost
```

Interpretation:

```txt
DecisionScore > 50:
  keep

10 to 50:
  keep but monitor

-10 to 10:
  partialize or sandbox

-50 to -10:
  replace or merge

< -50:
  abandon/quarantine
```

# 18. Safety Rules

Automatic optimization must have guardrails.

Never auto-replace without review when package is:

```txt
- crypto
- auth
- payment
- database driver
- compression/archive
- parser for security-sensitive input
- date/time with timezone complexity
- internationalization
- legal/compliance related
- native system interface
```

For those:

```txt
Default action:
  keep, sandbox, vendor, or human review
```

Auto-inline is safe mostly for:

```txt
- tiny pure functions
- formatting helpers
- simple string/array utilities
- local-only transformations
- dead packages
- duplicate internal helpers
```

# 19. Why CodeGraph Makes This Better Than npm

npm can tell:

```txt
You installed package X.
```

Bundlers can sometimes tell:

```txt
You imported function Y.
```

But RsTs can tell:

```txt
You import function Y.
Y uses helper A and B.
Helper B pulls dependency Z.
But your input types never trigger path B.
Therefore Z is not needed in production graph.
```

That is graph intelligence.

Also:

```txt
Package update adds new function.
Function not imported.
But package top-level init now runs telemetry.
Risk increased.
```

This is exactly the kind of thing that line-based tools miss.
